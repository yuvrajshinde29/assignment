		creating indexes:
		1.	primary key (default index):
			
		2. unique index:
			CREATE UNIQUE INDEX index_name on table_name(col_name)
			
		3. composite index:
			index on two or more columns together.
			store index entries in a sorted order on order of columns in the index.
			
				CREATE INDEX index_name on table_name(col_name1, col_name2);
				
				Left most prefix rule:
					mysql can use the index only for filters starting from the leftmost columns from the index
					
					if create :
						CREATE INDEX idx_name_city ON users(first_name, last_name);
					then MySQL can use this index for
						WHERE first_name = 'Yuvraj'	✅ Yes
						WHERE first_name = 'Yuvraj' AND last_name = 'Shinde'	✅ Yes
						WHERE last_name = 'Shinde'	❌ No
			
		4. prefix index
			prefix index created for string columns where first n characters(a prefix) of column value are indexed insted of entire value. 
				-save space for longer column values
				
				CREATE INDEX index_name on table_name(column_name(N))
				
				N- number of character to index.
				
				when use- when column is too long varchar(255) or TEXT

		fulltext indexes:
			it use for fast text searching within large text-based columns such as char,varchar,text
			database builds  a special index on words in text column(s)
			common words like is, the, and are ignored.
			return results ranked by relevence.
			
			creating full-text index:
				1. during table creation:
					create table article(
						int id,
						title varchar(200),
						body text,
						FULLTEXT (title,body)
					)
					
				2. on an existing table:
					create FULLTEXT index index_name on table_name(col_name1, col_name2);
					
			Quering fulltext indexes:
				select * from articles 
				where MATCH(title, body)AGAINST('mysql totorial')
				
				MATCH(title, body) → columns to search
				AGAINST('mysql tutorial') → search string
				Returns rows ranked by relevance automatically
				
			search mode:
				1. natural language mode(default)
					select * from articles
					where MATCH(title, body) AGAINST('mysql totorial')
					
			boolean mode:
				select * from articles 
				where MATCH(title, body)AGAINST('+mysql -totorial' IN BOOLEAN MODE)
				
				+mysql	  → must include
				-totorial → must not include
				
			search exact string:
				select * from articles 
				where MATCH(title, body)AGAINST('"mysql totorial"' IN BOOLEAN MODE )

	Transactions:
		are sequence of one or more SQL operations treated as single unit.
		
		transaction follow acid properties.
		| Property          | Meaning                                                       |
		| ----------------- | ------------------------------------------------------------- |
		| A – Atomicity     | All operations succeed or none (all-or-nothing)               |
		| C – Consistency   | Database remains in a consistent state after transaction      |
		| I – Isolation     | Each transaction executes independently of others             |
		| D – Durability    | Once committed, changes are permanent, even if system crashes |
		
		1. START TRANSACTION;
			or
			BEGIN;
			
		2. COMMIT;	--save all changes permanently 
		
		3. ROLLBACK;	--undoes all changes since transaction started
		
		4. SAVEPOINT; 	--make a point inside a transaction to roll back to
			SAVEPOINT s1;
			--rollback to this point
			ROLLBACK TO s1;
			
		most databases (like MySQL) have auto-commit on so to use transaction disable auto-commit
		SET autocommit = 0;

		Transaction isolation levels:
			Isolation determines how/when changes by one transaction are visible to others
			| Level            | Description                           | Dirty Read | Non-Repeatable Read | Phantom Read |
			| ---------------- | ------------------------------------- | ---------- | ------------------- | ------------ |
			| READ UNCOMMITTED | Can see uncommitted changes           | ✅          | ✅                   | ✅            |
			| READ COMMITTED   | Only committed changes                | ❌          | ✅                   | ✅            |
			| REPEATABLE READ  | Same rows consistent in a transaction | ❌          | ❌                   | ✅            |
			| SERIALIZABLE     | Full isolation                        | ❌          | ❌                   | ❌            |
			
			Dirty read:
				dirty read happen when a transaction reads data that another transaction has modified but not commited yet
				If the other transaction rolls back, the data you read becomes invalid.
			Non-Repeatable Read:
				A non-repeatable read happens when a transaction reads the same row twice, but gets different values because another transaction modified and committed it in between.
				
			Phantom Read:
				A phantom read happens when a transaction reads a set of rows, and another transaction inserts or deletes rows that meet the same query condition.
				When the first transaction reads again, it sees new “phantom” rows.
			----
			How set Transaction isolation Level:
			Globally (for the entire server)
				-SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
			For the current session
				SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
			For a single transaction
				SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
				START TRANSACTION;

		-----------
		Deadlock:
			When two transactions each lock different rows or tables, and then try to access the other’s lock.
