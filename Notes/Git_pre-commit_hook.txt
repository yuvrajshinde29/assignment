Git Pre Commit (Coding Convention):
----------------------------------------------
	A Git pre-commit hook for enforcing coding conventions is a script that runs automatically before a commit is finalized
	ensuring your code follows certain standards (formatting, linting, naming, etc.) before being saved in the repository.
		
		-A linter is a tool that automatically checks your source code for errors, style issues, and potential bugs — before you even run your program
		eslint -Enforces JavaScript/TypeScript rules and conventions
		
	A pre-commit hook is a script placed in the .git/hooks/ directory.
		It runs before git commit — if it exits with a non-zero status, the commit is aborted.
		
		This is perfect for:
			Enforcing coding style (e.g., PEP8, ESLint rules)
			Checking file formatting
			Preventing bad commit messages
			Blocking large files or secrets
		
	what pre-commit hooks do:
		-inspect code changes staged for commit.
		-Run linters, formatters, unit tests, or other code quality checks.
		-Abort the commit if any issue are found forcing corrections before the commit proceeds.
		-Enforce team coding standards consistently across all contributors.
	-------------------------------------------
	Husky:
		Husky is a tool that connects scripts (like ESLint) to Git hooks.

		Git hooks are small scripts that run automatically before or after certain Git actions — e.g., commit, push, etc.

| File Extension | Package `"type"`               | Module Type |
| -------------- | ------------------------------ | ----------- |
| `.js`          | `"type": "commonjs"` (default) | CommonJS    |
| `.js`          | `"type": "module"`             | ESM         |
| `.mjs`         | any                            | ESM         |
| `.cjs`         | any                            | CommonJS    |



Import dependencies
const js = require('@eslint/js');
const globals = require('globals');


@eslint/js → Provides ESLint JavaScript recommended configurations.

globals → Provides predefined global variables for different environments (browser, Node, etc).

2️⃣ Export configuration
module.exports = [ ... ];


ESLint reads the exported array of configuration objects.

Using CommonJS (module.exports) because the file is .cjs.

3️⃣ Target files
files: ['**/*.{js,mjs,cjs}'],


The ESLint rules in this block will apply to all JavaScript files, including:

.js → standard JavaScript

.mjs → ES modules

.cjs → CommonJS modules

**/* → recursively includes all folders.

4️⃣ Recommended ESLint config
...js.configs.recommended


Spreads the recommended ESLint rules from @eslint/js.

Gives a good default set of rules for modern JS without having to write them all manually.

5️⃣ Language options
languageOptions: {
  globals: globals.browser,
  sourceType: 'commonjs',
},


globals → Defines predefined global variables that ESLint should recognize.

globals.browser → Recognizes window, document, etc., as valid.

sourceType: 'commonjs' → Tells ESLint that these files use CommonJS syntax (require, module.exports).

Without this, ESLint might assume ES modules and throw errors on require.

6️⃣ Custom rules
rules: {
  'no-unused-vars': 'warn',
  'no-console': 'warn',
  semi: ['error', 'always'],
  quotes: ['error', 'single'],
  'no-multi-spaces': 'error',
},


no-unused-vars: 'warn' 		→ Warns if variables are declared but not used.
no-console: 'warn' 			→ Warns when console.log() is used.
semi: ['error', 'always'] 	→ Enforces semicolons at the end of statements.
quotes: ['error', 'single'] → Enforces single quotes for strings.
no-multi-spaces: 'error'	→ Disallows multiple spaces (keeps code tidy).
